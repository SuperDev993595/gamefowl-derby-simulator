# Gamefowl Derby Simulator — Full Implementation Guide

This guide explains how to implement the project from scratch. It does not contain code; it gives architecture, stack choices, phases, and step-by-step directions. Use it together with **PROJECT_ANALYSIS.md**.

---

## Part 1: Tech Stack and Architecture

### 1.1 Recommended Stack (Scalable, Maintainable)

| Layer | Recommendation | Why |
|-------|----------------|-----|
| **Frontend** | React or Next.js (or Vue/Svelte if you prefer) | Component-based, good for real-time UI and forms; Next.js helps with SSR/SEO and API routes if you want a single repo. |
| **Backend API** | Node.js (Express/Fastify) or .NET Core | Fits “lightweight, scalable”; Node aligns with a JS/TS full stack. For “thousands concurrent,” use stateless API + background workers. |
| **Real-time** | WebSockets (Socket.io or native WS) or Server-Sent Events | For live bracket/match updates and admin dashboard. |
| **Database** | PostgreSQL (primary) + Redis (cache/sessions/queues) | PostgreSQL for tournaments, entries, users, matches; Redis for session store, rate limiting, and optionally job queues. |
| **Auth** | OAuth 2.0 / OIDC (e.g. Auth0, Clerk, Supabase Auth, or Keycloak) | SSO without storing passwords; token refresh and CSRF are handled by the provider or your middleware. |
| **Hosting** | Single cloud (e.g. AWS, GCP, or Vercel + DB) or Railway/Render | Use a managed DB and optional managed Redis; run API and workers on the same provider for simplicity. |

You can swap any layer (e.g. Django + DRF instead of Node) as long as you keep: stateless API, background jobs for round processing, and a real-time channel.

### 1.2 High-Level Architecture

- **Client (browser):** SPA or SSR app. No plugins; works on Chrome, Firefox, Safari, Edge.
- **API server:** REST (or GraphQL) for CRUD; one real-time channel (WebSocket or SSE) for bracket/match updates.
- **Auth provider:** Handles login; your API validates tokens and creates/links a **user** with an autogenerated **username** (e.g. `player_abc123`).
- **Database:** Stores users, tournaments, entries, matches, rounds, and (if needed) token balances and goals.
- **Background worker / cron:** Runs “round advancement”: pair active entries, resolve matches, update standings, detect eliminations and tie-breakers. Keeps the API fast under load.
- **Admin:** Same app with role check, or separate dashboard; uses same API + real-time.

### 1.3 Data You Need Up Front

- **Static:** Breeds (20), derby types (4), trait ratings per (breed, derby type). Convert TRAIT RATINGS.xlsx to JSON or DB seed once (see PROJECT_ANALYSIS and Part 3).
- **Breed images:** Copy or symlink `resources/breed pics` into the app’s static/public folder; map breed id to filename (handle GRAY/GREY and THOMPSON WHITE as in PROJECT_ANALYSIS).

## Part 2: Implementation Phases (Order of Work)

Do these in order so each phase has a working base for the next.

### Phase 1: Foundation (Repo, Data, Auth)

1. **Repository and build**
   - Create repo structure: e.g. `client/`, `server/`, `shared/` (types/constants).
   - Add dependency management and build scripts (e.g. npm workspaces or separate package.json per app).
   - Document how to install deps and run dev (build instructions).

2. **Seed data**
   - Convert TRAIT RATINGS.xlsx to a seed file (JSON or SQL). Structure: one record per (breed, derby type) with power, speed, intelligence, stamina, accuracy. Fix REDQUILL Long Heel (four values → add default accuracy).
   - Add breed list and image filename mapping (including GRAY/GREY, THOMPSON WHITE).
   - Load seed data on first deploy or via a one-off script.

3. **Database schema (conceptual)**
   - **users:** id, external_id (from SSO), email (optional), username (autogenerated), created_at.
   - **tournaments:** id, name, derby_type, total_rounds (10–20), start_at, status (draft/scheduled/running/finished), created_at, config (JSON if needed).
   - **entries:** id, tournament_id, user_id, breed_id (or breed name), token_cost_paid, status (active/eliminated), wins, losses, created_at.
   - **matches:** id, tournament_id, round_number, entry_a_id, entry_b_id, winner_entry_id (nullable), status (pending/played), played_at.
   - **rounds:** optional; or derive from matches (round_number).
   - **tie_breaker_matches:** same shape as matches, or a flag on matches (is_tie_breaker).
   - **token_balances:** user_id, balance, updated_at (if you manage tokens in your DB).
   - **breeds / breed_traits:** id, name, image_filename; and traits per derby_type (from seed).

4. **SSO integration**
   - Integrate your chosen provider (Auth0, Clerk, etc.): login redirect, callback, token validation.
   - On first login: create user and assign autogenerated username (e.g. from provider id or random).
   - Ensure token expiration and refresh are handled; use HTTP-only cookies or short-lived access tokens + refresh flow. Protect API routes with middleware that validates tokens and attaches user.
   - Apply CSRF protection for state-changing requests (provider docs or same-site cookies + CSRF tokens).

### Phase 2: Tournament and Entry Logic

5. **Tournament CRUD (admin)**
   - Create/update tournament: name, derby_type, total_rounds (10–20), start_at.
   - List tournaments; filter by status. No code change needed for “restart” or “clone”: implement as “create new tournament” with same config (clone) or “reset and run again” (restart) using the same schema.

6. **Entry and registration**
   - Player flow: choose tournament → choose derby (already set on tournament) → select breed(s) — clarify spec: one bird per entry or multiple; usually one entry = one bird/breed per tournament.
   - Deduct entry cost (tokens); create **entry** linked to user and tournament. Enforce: registration only before tournament start; optional max entries per user per tournament.

7. **Round and pairing rules**
   - **Round 1:** Pair entries arbitrarily (e.g. random or order) into matches. Each match has two entries.
   - **Round N (N > 1):** Only **active** entries (not eliminated) are paired. Pair only winners from the previous round with each other (winners vs winners). If odd count, one gets a bye (or handle by rule: e.g. random bye).
   - **Elimination:** When an entry has so many losses that it can no longer reach first place (e.g. mathematically eliminated), set status to eliminated and do not include in future pairings.
   - **Scheduled rounds:** Run exactly `total_rounds` rounds (e.g. 10). After that, compute standings (e.g. by wins, then losses).

8. **Match resolution**
   - For each match, determine winner using trait ratings for the tournament’s derby type. Options: (a) weighted sum of traits + randomness, (b) dice/roll per trait with modifiers, (c) deterministic formula with seed. Use the same trait data from your seed. Store winner_entry_id and mark match as played.

### Phase 3: Tie-Breakers and Winner

9. **Tie detection**
   - After the last scheduled round, rank entries by wins (desc), then losses (asc). If the top group has more than one entry with the same (wins, losses), they are tied for first.

10. **Tie-breaker matches**
    - While there is more than one tied-for-first entry: run additional rounds. Only the tied entries participate; pair them (winners vs winners) until one remains. That one is the sole winner. Apply the same elimination rule if you allow “elimination within tie-breaker” by losses, or keep it simple: only the top tier, single-elimination style until one winner.

11. **Payouts and state**
    - Award tokens (and bonuses) to the winner; optionally to top 3. Mark tournament as finished. Persist final standings for leaderboards and history.

### Phase 4: Real-Time and Admin UX

12. **Real-time updates**
    - When matches are resolved or rounds advance, publish events (e.g. “round_completed”, “match_result”, “standings_updated”). Clients subscribed to the tournament (or global) channel update the bracket view or match list without refresh. Use WebSockets or SSE from the server; trigger from the worker or API after each batch of match results.

13. **Admin dashboard**
    - UI: create/edit tournament (name, rounds, start time, derby type). Buttons: Start tournament (change status to running), Restart (reset entries/matches and re-run or create clone), Clone (new tournament with same config). Real-time view: bracket or table of matches and standings that updates live.

14. **Player bracket view**
    - Public or authenticated view: list of matches per round, results, and current standings. Same real-time subscription so it updates as results come in.

### Phase 5: Modes, Economy, and Polish

15. **Token economy**
    - Define starting balance (e.g. for new users). Entry cost per tournament (fixed or per derby type). Winner payout (e.g. pool share or fixed + bonus). Store balances and transactions (entry fee, payout) so you can audit and support “pay to enter.”

16. **Quick Play**
    - Flow: land on app → sign in (SSO) → choose derby type (or “next available” tournament) → select breed → pay tokens → enter. Minimal profile (just username and balance). No long-term goals.

17. **Breeders Play**
    - Profile: username, stats (entries, wins, top placements). Goals: e.g. “Win 3 Long Heel derbies,” “Enter 10 derbies.” Track progress in DB and show in UI. Leaderboards: by wins, by earnings, or by goals completed. Reuse same entry and tournament logic; only the presentation and goals layer differ.

18. **Guest play (optional)**
    - If product wants “play without account,” allow a limited flow: e.g. play with fake balance and no persistence, or create an anonymous session that can be converted to an account later. Otherwise, require SSO for all entry.

### Phase 6: Deployment and Docs

19. **Deployment**
    - Deploy API and worker to your chosen cloud (e.g. one service for API, one for worker; or same process with a worker thread). Use managed PostgreSQL and Redis. Point frontend to API URL; use env vars for auth provider keys and DB URLs. Ensure HTTPS and secure cookies.

20. **Build and run documentation**
    - README: install deps, env vars required, how to run dev (client + server + DB + Redis), how to run tests, how to build for production. Add a short “Deploy” section (e.g. Docker or platform-specific steps).

21. **User guide**
    - Document: how to sign up / sign in (SSO), how to join a derby (choose derby, select bird, pay tokens), how tie-breakers work (multiple winners → extra matches until one winner), and (for admins) how to create/start/restart/clone tournaments and read the real-time dashboard.

## Part 3: Critical Design Decisions (Checklist)

- **One entry = one bird (breed) per tournament:** Confirm; then entry has one breed_id and one set of trait ratings for the tournament’s derby type.
- **Pairing rule when odd number of winners:** Bye (one entry gets a free win) or another rule; document it.
- **Elimination rule:** e.g. “If an entry has more than X losses and cannot catch the current best, mark eliminated.” X depends on total_rounds; implement a small function that, given (wins, losses, total_rounds, current_leader_wins), returns whether the entry is eliminated.
- **Match resolution formula:** Choose one formula and stick to it; test with trait data so outcomes feel fair (no breed always wins).
- **Token source:** Tokens only in-app (you control balance) or real money? If real money, you need payments integration and compliance; the guide assumes in-app only.
- **Restart vs clone:** “Restart” = same tournament id, clear entries/matches and set status back to draft/scheduled. “Clone” = new tournament id, copy config. Implement both without code changes by using the same API and schema.

## Part 4: Acceptance Criteria Mapping

Use this to verify you’re done:

| Criterion | Where it’s implemented |
|-----------|------------------------|
| Unlimited registrants, no performance drop | Efficient pairing (batch or queue), DB indexes on tournament_id, entry status; load test. |
| Round generation and elimination 100% correct | Phase 2 (pairing + elimination logic); unit tests with fixed entry lists and rounds. |
| Tie-breakers until one winner | Phase 3 (tie detection + tie-breaker matches); test with 2–3 tied entries. |
| SSO, token expiration, CSRF | Phase 1 (auth provider + API middleware); security review or checklist. |
| No browser plugins; works on major browsers | Phase 4–5 (frontend): no plugins, standard HTTP/WS; test on Chrome, Firefox, Safari, Edge. |
| Admin: configure name, rounds, start | Phase 2 + 4 (tournament CRUD + admin UI). |
| Real-time bracket/match list | Phase 4 (real-time channel + UI). |
| Restart/clone without code change | Phase 2 (API and schema support both). |

## Part 5: Suggested Order of Implementation (Summary)

1. Repo, build, and seed data (breeds, traits, images).  
2. DB schema and migrations.  
3. SSO and user creation with autogenerated username.  
4. Tournament CRUD (admin).  
5. Entry registration and token deduction.  
6. Round pairing (winners vs winners) and elimination rule.  
7. Match resolution using traits.  
8. Tie detection and tie-breaker rounds.  
9. Payouts and tournament completion.  
10. Real-time channel and UI updates (admin + player).  
11. Quick Play and Breeders Play flows and goals.  
12. Deployment, README, and user guide.

This order keeps each step testable and avoids big rewrites. Use **PROJECT_ANALYSIS.md** for data details and **this guide** for how to implement the system end to end.
