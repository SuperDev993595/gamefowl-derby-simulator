# Implementation Verification & Run Guide

This document (1) verifies what is fully implemented against the spec, (2) notes any gaps, and (3) gives you a step-by-step guide to run and use the app (no code changes).

---

## Part 1: Implementation Verification

### ✅ Fully implemented

| Requirement | Status | Where |
|-------------|--------|--------|
| **Browser-based, no plugins** | ✅ | React SPA (Vite); works in Chrome, Firefox, Safari, Edge. |
| **Unlimited entrants per tournament** | ✅ | No hard cap; entries stored per tournament; pairing handles any count (with bye when odd). |
| **Single bracket per entrant** | ✅ | One entry = one user + one breed per tournament. |
| **Configurable rounds (10–20)** | ✅ | Admin sets 10–20 when creating tournament; enforced in backend. |
| **Winners advance; pair winners vs winners** | ✅ | `tournament_engine.py`: round 1 = all active entries; round N = winners from round N−1 only. |
| **Eliminated entries get no further pairings** | ✅ | `mark_eliminated()` + `get_entries_for_round()` only uses active entries / previous-round winners. |
| **Tie-breakers until one winner** | ✅ | After last scheduled round, if tied leaders: `is_tie_breaker` phase runs extra rounds among tied entries until one winner. |
| **Autogenerated username** | ✅ | Register returns `player_xxxx`-style username; admin seed is `admin`. |
| **Admin: tournament name, rounds, start** | ✅ | Admin UI: create tournament (name, derby type, 10–20 rounds); start date optional in API. |
| **Real-time bracket / match list** | ✅ | WebSocket `/ws/tournaments/{id}`; admin “Advance round” triggers broadcast; Bracket page subscribes and refetches. |
| **Restart tournament** | ✅ | Admin “Restart”: clears matches, resets entry wins/losses/status to active, tournament back to draft. |
| **Clone tournament** | ✅ | Admin “Clone”: new tournament with same name (Clone), derby type, rounds, start_at. |
| **Token entry cost & winner payout** | ✅ | Entry deducts `ENTRY_COST_TOKENS`; on tournament finish winner gets pool (entries × cost) + `WINNER_BONUS_TOKENS`. |
| **20 breeds, 4 derby types, trait ratings** | ✅ | Seed from TRAIT RATINGS.xlsx (long/short heel, long/short blade); REDQUILL Long Heel missing accuracy defaulted. |
| **Breed images** | ✅ | Served at `/breed-pics/` from `resources/breed pics/`; Enter page shows grid with images. |
| **Match resolution from traits** | ✅ | Weighted sum (power, speed, intel, stamina, accuracy) + randomness per match. |
| **Source code + build instructions** | ✅ | README: backend (uvicorn) and frontend (npm run dev / npm run build). |
| **Short user guide** | ✅ | README “User guide (short)” + this doc. |

### ⚠️ Intentional differences / not implemented

| Topic | Spec / doc | Current implementation | What you might do later |
|-------|------------|-------------------------|--------------------------|
| **Auth** | SSO only (OAuth/OIDC) | JWT with email/password (register + login) | Add Auth0/Clerk/Supabase and swap login flow; keep “autogenerated username” on first SSO login. |
| **Start date/time** | Admin sets start | API supports `start_at`; admin UI does not expose it in create form | Add a start date/time field in Admin create/edit if you want to enforce “registration until start”. |
| **Quick Play vs Breeders Play** | Two modes | Single flow: sign in → derbies → enter; no separate “Quick” vs “Breeders” | Add a “Breeders” profile page with goals/achievements and reuse same entry/tournament logic. |
| **Guest play** | Optional | Not implemented; must register/sign in to enter | If desired, add anonymous sessions or “play without account” with limited persistence. |
| **Deployed version** | Deliverable | Not deployed; run locally only | Deploy backend (e.g. Railway/Render) + frontend (e.g. Vercel/Netlify) and set `VITE_API_URL` + `DATABASE_URL`. |
| **Database** | PostgreSQL + Redis | SQLite (dev); no Redis | For production: set `DATABASE_URL=postgresql+asyncpg://...`; add Redis only if you need cache/queues. |

### Summary

- **Tournament logic (pairing, elimination, tie-breakers, payouts)** and **admin/player flows** are fully implemented and match the spec.
- **Auth** is JWT (email/password) instead of SSO; the rest (username, tokens, entry, bracket) works as specified.
- **Real-time** works via WebSocket when admin advances rounds.
- **Build and run** are documented in the README; deployment and SSO/Breeders/guest are optional next steps.

## Part 2: Step-by-Step Run Guide

### Prerequisites

- **Python 3.10+** (e.g. 3.12) on PATH.
- **Node.js 18+** and **npm** (for the frontend).
- Two terminals (one for backend, one for frontend).

### Step 1: Backend

1. Open a terminal and go to the project root:
   - `cd D:\Workspace\freelancer\gamefowl-derby-simulator`
2. Go into the backend folder:
   - `cd backend`
3. (Optional but recommended) Create and activate a virtual environment:
   - `python -m venv venv`
   - Windows: `venv\Scripts\activate`
   - macOS/Linux: `source venv/bin/activate`
4. Install dependencies:
   - `pip install -r requirements.txt`
5. (Optional) Copy env example and edit if you need different settings:
   - Copy `.env.example` to `.env`
   - Change `SECRET_KEY` for production; for local dev you can leave defaults.
6. Start the API server:
   - `uvicorn app.main:app --reload --host 0.0.0.0 --port 8000`
7. Confirm:
   - In the terminal you should see startup and no errors.
   - Open **http://localhost:8000/docs** in a browser — the Swagger UI should load.
   - On first run, the app creates the DB and seeds breeds + default admin. You should see a line like “Seeded breeds, traits, and admin…” in the console (if you added a print in seed).

Default admin: **admin@example.com** / **admin123**.

### Step 2: Frontend

1. Open a **second** terminal and go to the project root again:
   - `cd D:\Workspace\freelancer\gamefowl-derby-simulator`
2. Go into the client folder:
   - `cd client`
3. Install dependencies:
   - `npm install`
4. (Optional) If your API is not at `http://localhost:8000`, create a `.env` in `client/` with:
   - `VITE_API_URL=http://localhost:8000`
5. Start the dev server:
   - `npm run dev`
6. Confirm:
   - Browser opens (or go to **http://localhost:5173**). You should see the “Gamefowl Derby” home page.

### Step 3: First-time setup (admin)

1. In the app, click **Sign in**.
2. Log in with **admin@example.com** / **admin123**.
3. You should be redirected to **Derbies** and see an **Admin** link (and your token balance).
4. Go to **Admin**.
5. Create a tournament:
   - Enter a name (e.g. “Spring Long Heel”).
   - Choose **Long Heel** (or any derby type).
   - Set rounds to **10** (or 10–20).
   - Click **Create**.
6. Do **not** click **Start** yet — you’ll want at least one other player to enter first.

### Step 4: Player flow (second user)

1. Open a **new browser window** (or incognito) so you’re not logged in as admin.
2. Go to **http://localhost:5173**.
3. Click **Register** and create a second account (e.g. `player@test.com` / `test123`).
4. After register you’ll be on Derbies with a token balance (e.g. 100).
5. Click **Enter this derby** on the tournament you created.
6. Pick a **breed** (rooster) and click **Pay & Enter**.
7. You’ll be taken to the **bracket** page; standings will show your entry. You can leave this tab open for real-time updates.

Repeat with more browsers/users if you want more entries.

### Step 5: Running the tournament (admin)

1. In the admin browser, go to **Admin** and find your tournament.
2. Click **Start**. Status should become “running”.
3. Click **Advance round**. The backend will:
   - Create round 1 pairings (and possibly a bye if entry count is odd).
   - Resolve each match using breed traits.
   - Update wins/losses and eliminate anyone who can’t catch the leader.
4. Keep clicking **Advance round** until:
   - You’ve run all scheduled rounds (e.g. 10), and
   - Either one winner is clear, or tie-breaker rounds run until one winner.
5. In the **player** window (bracket page), the standings and matches should update automatically when you advance (WebSocket + refetch).

### Step 6: After the tournament

- **Winner:** Gets (number of entries × entry cost) + winner bonus tokens (see backend `.env`: `ENTRY_COST_TOKENS`, `WINNER_BONUS_TOKENS`).
- **Restart:** In Admin, **Restart** clears all matches and resets all entries (wins/losses/status) so you can run the same tournament again. Entries stay; tokens are **not** refunded on restart.
- **Clone:** **Clone** creates a new tournament with the same config (name + “(Clone)”); you can then add new entries and start it.

### Step 7: Viewing bracket and standings anytime

- From **Derbies**, click **View bracket** on any tournament.
- You’ll see **Standings** (wins, losses, status) and **Matches** by round (including tie-breaker rounds if any). No need to refresh when admin advances — the page listens for WebSocket events and refetches.

## Part 4: If something doesn’t work

- **Backend won’t start:** Ensure Python 3.10+ and `pip install -r requirements.txt` completed. If you use a venv, activate it before running uvicorn.
- **Frontend can’t reach API:** Ensure backend is running on port 8000. If you use another host/port, set `VITE_API_URL` in `client/.env` and restart `npm run dev`.
- **CORS errors:** Backend `.env` (or default in `config.py`) has `CORS_ORIGINS`; include your frontend origin (e.g. `http://localhost:5173`).
- **No breeds / admin:** First run seeds the DB. If you deleted `derby.db`, remove it and restart the backend so seed runs again.
- **Bracket doesn’t update:** Ensure you’re on the bracket page for that tournament; WebSocket URL is built from the same host as the API (e.g. `ws://localhost:8000/ws/tournaments/1`). Check browser dev tools (Network → WS) for connection errors.

Using this, you can confirm the implementation and run the app end-to-end without changing code.
