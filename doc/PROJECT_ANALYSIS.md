# Gamefowl Derby Simulator — Project Analysis

This document summarizes the project requirements, client vision, and the provided assets (breeds doc, trait ratings, breed images) so the implementation stays aligned with the spec.

---

## 1. Project Requirements (Original Spec)

### Tournament system
- **Browser-based** sports simulation; no plugins (Chrome, Firefox, Safari, Edge).
- **Unlimited entrants** per tournament; each entrant in a single bracket.
- **Rounds:** 10–20 per tournament (configurable).
- **Pairing:** Each round, active entries are matched (winners vs winners); eliminated entries get no further pairings.
- **Tie-breakers:** After scheduled rounds, if two or more are tied for first, run extra tie-breaker matches until one winner.

### Player access
- **SSO only** — no local passwords; token expiration, CSRF, etc. must be handled.
- **Autogenerated username** after sign-in for leaderboards and match cards.

### Admin
- **Dashboard:** Tournament name, number of rounds (10–20), start date/time.
- **Real-time** bracket or match list as results are processed.
- **Restart or clone** tournaments without code changes.
- **Back-end:** Secure, scalable for thousands of concurrent users, easy to maintain.

### Deliverables
1. Source code and build instructions.
2. Deployed version (your hosting or recommended cloud).
3. User guide: player sign-up, admin controls, tie-breaker logic.

---

## 2. Client Vision (Layman’s Terms)

### Core loop
- User picks **roosters** (breeds) and enters a **derby** of their choice.
- **Derby types:** Long Heel, Short Heel, Long Blade, Short Blade.
- User **selects roosters**, **pays tokens**, and **enters** the derby.
- **10 matches** per derby (fixed in client description; spec says 10–20 configurable — recommend making this configurable with 10 as default).
- **Winner** receives more tokens and bonuses.
- **Tie-breakers** used when multiple winners until a single winner.

### Play modes
- **Quick Play:** Choose derby + birds and enter (no account/profile required if SSO is still the gate; clarify if “guest” is allowed).
- **Breeders Play:** Create a profile and play toward **goals** (e.g. achievements, progression, leaderboards).

## 4. TRAIT RATINGS.xlsx — Summary

### Structure
- **Rows:** One per breed (same 20 as above).
- **Columns:** Four derby types — **Long Heel**, **Short Heel**, **Long Blade**, **Short Blade**.
- **Cell format:** Each cell is a string of five numbers: **Power.Speed.Intel.Stamina.Accuracy** (e.g. `10.5.5.8.6`).

### Example (first few breeds)

| Breed      | Long Heel   | Short Heel  | Long Blade  | Short Blade |
|------------|-------------|-------------|-------------|-------------|
| HATCH      | 10.5.5.8.6  | (see sheet) | (see sheet) | (see sheet) |
| KELSO      | 7.7.6.5.8   | …           | …          | …          |
| CLARET     | 5.8.6.6.7   | …           | …          | …          |
| ROUNDHEAD  | 5.8.9.6.8   | …           | …          | …          |
| …          | …           | …           | …          | …          |

- One cell (REDQUILL Long Heel) has only four numbers: `4.10.6.8` — treat as Power=4, Speed=10, Intel=6, Stamina=8, **Accuracy=missing**; in code use a default (e.g. 5) or flag for data fix.
- **Use in game:** For a given derby type, each breed has a fixed vector (P, S, I, St, A). Match resolution can use these (e.g. weighted comparison, dice + modifiers, or full sim) depending on design.

### Implementation notes
- **Data source:** Either import the xlsx at build time or convert to JSON/DB seed so the server reads structured trait arrays per (breed, derby type).
- **Normalize breed names** between doc (e.g. “BROWNRED”) and assets (“BROWN RED.jpg”) for lookups.

## 7. Recommended Next Steps

1. **Data pipeline**
   - Convert **TRAIT RATINGS.xlsx** to a single JSON (or DB seed) with structure:  
     `{ breedId, breedName, derbyType, power, speed, intelligence, stamina, accuracy }`  
     (or one object per breed with derby types as keys).
   - Fix the one incomplete rating (e.g. REDQUILL Long Heel `4.10.6.8` → add Accuracy default or correct in source).
   - Normalize breed keys for **breed pics** (e.g. “BROWNRED” → “BROWN RED” for file lookup).

2. **Match resolution**
   - Define how trait ratings affect outcome (e.g. formula: weighted sum + randomness, or full sim step-by-step). Use the same trait set from the spreadsheet.

3. **Economy**
   - Define entry cost (tokens), winner payout, and bonuses; support both Quick Play and Breeders Play if both use tokens.

4. **Modes**
   - **Quick Play:** Minimal flow (derby type → select birds → pay → enter); optional guest vs “must SSO”.
   - **Breeders Play:** Profile, goals, and progression; reuse same derby and trait data.

5. **Assets**
   - Serve breed images from `resources/breed pics` (or copied into app’s public/static folder) and map breed id/name to filename (including GRAY/GREY and THOMPSON WHITE).

This analysis gives a single reference for scope, data, and assets for the gamefowl derby simulator.
